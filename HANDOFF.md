## 1. 프로젝트 현재 상태

- **순수 바닐라 JS 단일 파일 게임** (`game.js` 하나, 빌드 없음, 의존성 없음).
- Kenney Pixel Platformer 에셋 기반 횡스크롤 플랫포머. 480×270 논리 해상도, 18px 타일.
- **총 50개 스테이지**:
  - 스테이지 1~3: 수제 레벨 (`buildLevel1/2/3` — 초원/사막/산)
  - 스테이지 4~50: 절차 생성 (`buildGeneratedLevel(stage)` — 시드 고정이라 항상 같은 지형)
- 마지막 커밋: `59db7a3` "Add stages 4-50 via seeded procedural level generation"
- 리모트: https://github.com/gridnflow/pixel-adventure-game (main 브랜치)

### 실행 방법
```bash
cd pixel-adventure-game
python3 -m http.server 8080   # 8000 포트는 이 머신에서 다른 프로세스가 쓰는 경우가 있음
open http://localhost:8080
```
- 특정 스테이지 바로 시작(개발용 치트): `http://localhost:8080#dev37` / `#dev50x100` (50스테이지, 타일 X=100 지점)

## 2. 최근 작업 요약 (스테이지 4~50 절차 생성)

`game.js`의 `buildLevel3()` 뒤, `LEVELS` 배열 정의 앞에 생성기 코드가 있다.

### 구조
- `mulberry32(seed)`: 시드 고정 PRNG. **시드 = `stage * 1013904223 + 5`**.
- 세그먼트 함수 6종을 커서 `x`를 옮기며 이어 붙임:
  | 함수 | 내용 | 뒤에 지면 강제? |
  |---|---|---|
  | `segFlat` | 평지 + 코인/적/상자/장식/파워업 | – |
  | `segPit` | 구덩이 (4타일 이상이면 중간 발판 자동) | ✅ `needGround` |
  | `segStones` | 징검다리 섬 2~3개 | – (지면으로 끝남) |
  | `segLadder` | 구덩이 위 발판 사다리 + 보석 도전 | ✅ |
  | `segPlateau` | 발판 상승 → 고지대(top 7) → 낙하 | ✅ |
  | `segHill` | 지면 위 계단 블록 | – |
- `needGround` 플래그: 구덩이류 세그먼트 직후에는 반드시 `segFlat`이 와서 착지 지점을 보장한다. **이 로직을 깨면 클리어 불가능 맵이 나올 수 있다.**
- 난이도 `d = min(1, (stage - 3) / 40)`: 적 수·톱니 비율·구덩이 폭·맵 길이(최대 250타일)에 반영.
- 테마 순환: `['green','desert','mountain'][(stage - 1) % 3]`.
- 보석: `state.gem` 플래그로 레벨당 최소 1개 보장 (사다리/고지대에서 못 나오면 골인 구간에 배치).

### 함께 바뀐 게임플레이 결정
- **`nextLevel()`에서 하트를 3으로 회복**시킨다. 원래는 전체 러닝 동안 하트 3개였는데 50스테이지 완주가 불가능해서 바꿨다. 밸런스 재조정 시 이 지점을 참고할 것.

## 3. 반드시 지켜야 할 제약 (함정 목록)

1. **점프 물리 한계** — 레벨 생성/수제 레벨 수정 시 절대 기준:
   - `GRAV=830, MOVE=130, JUMP_V=292` → 단일 점프: **가로 최대 ~5타일, 상승 최대 ~2.5타일**
   - 더블 점프(기본 1회) 포함 시 가로 ~9타일까지 가능하지만, **생성 레벨은 단일 점프 기준으로만 설계**했다 (더블 점프는 여유분).
   - 수제 레벨 3의 51~57 구덩이(7타일)는 더블 점프 전제의 의도적 설계다. 버그 아님.
2. **PRNG 호출 순서가 곧 맵이다.** 세그먼트 함수 안에서 `rng()` 호출을 추가/삭제/순서 변경하면 **기존 4~50 스테이지 지형이 전부 바뀐다.** 지형 유지가 필요하면 새 기능은 별도 `mulberry32` 인스턴스(예: 시드에 상수 오프셋)를 쓸 것. 유저 진행 저장 기능이 들어가면 특히 조심.
3. **그리드 기호 의미** (`isSolid`는 `# X ! x`만 충돌):
   - `=`는 일방통행 발판(아래서 통과, 위에서만 착지). `d`는 문, `o` 코인, `*` 보석, `S/D` 파워업, `s/t/c/m` 장식(비충돌).
4. **적 배치 y좌표**: `L.enemy(type, tx, ty)`의 `ty`는 "발이 닿는 지면 top - 1". 예: 지면 top이 12면 `ty=11`, 고지대 top 7이면 `ty=6`. `loadLevel()`에서 `(ty+1)*TS - box.h`로 변환된다.
5. **슬라임/톱니는 낭떠러지에서 반전**하는 AI라 좁은 섬에 배치해도 떨어지지 않지만, **폭 2타일 미만 지형에는 배치하지 말 것** (반전 판정이 떨린다).
6. **`LEVELS`는 팩토리 배열**이다 (`buildGeneratedLevel.bind(null, s)`). 레벨 로드는 항상 `LEVELS[idx]()` 호출 → 매번 새로 빌드되므로 그리드를 파괴적으로 수정해도(코인 수집 등) 재시작 시 복원된다.

## 4. 검증 방법 (이미 검증된 절차)

### 4-1. 전체 스테이지 정합성 검사 (Node, 브라우저 불필요)
```bash
node tools/test-levels.js   # "all stages OK" + exit 0 이어야 함
```
- `tools/test-levels.js`가 vm 샌드박스에 `document/Image/Audio/window` 스텁을 넣고 `game.js`를 로드한 뒤 50개 레벨을 전부 빌드해 검사한다.
- 검사 항목: ① 문(`d`) 존재 ② 스폰 지점 아래 지면 ③ 발 디딜 곳 없는 연속 세로줄(구덩이 폭) ≤ 3.
- **수제 스테이지 3의 gap run 7은 더블 점프 설계라 예외 처리되어 있다** (`GAP_EXCEPTIONS`). 새 의도적 예외가 생기면 그 테이블에 추가할 것.
- **레벨·생성기·물리를 건드리는 모든 작업 전후로 반드시 실행할 것.**

### 4-2. 렌더링 스모크 테스트 (헤드리스 Chrome)
```bash
"/Applications/Google Chrome.app/Contents/MacOS/Google Chrome" --headless --disable-gpu \
  --window-size=980,600 --virtual-time-budget=4000 \
  --screenshot=/tmp/stage.png "http://localhost:8080/#dev37"
```
스크린샷에서 HUD의 `LEVEL 37/50`, 지형, 적이 보이면 정상. **빈 화면이면 로드 실패다.**

## 5. 남은 작업 (우선순위 순)

### 5-1. [P1] 레벨 검증 강화 (기본 스크립트는 이미 `tools/test-levels.js`로 포함됨)
- "시뮬레이션 봇"(우측 이동+점프 그리디)으로 실제 클리어 가능성까지 검증하는 단계를 추가하면 이상적. 현재 검사는 구덩이 폭 기반 휴리스틱이라 수직 배치의 극단 케이스는 못 잡는다.
- README에 `node tools/test-levels.js` 실행법 한 줄 추가.

### 5-2. [P1] 진행 저장 (localStorage)
- 저장 항목: 도달한 최고 스테이지, 스테이지별 클리어 여부/최단 시간, 누적 최고 스코어.
- 키 제안: `pixelAdventure.v1` 하나에 JSON으로.
- 타이틀 화면에 "이어하기 (스테이지 N)" 표시. Enter=이어하기, 다른 키(예: N)=처음부터.
- 주의: 함정 #2 — 생성 레벨 지형이 바뀌면 저장된 기록의 의미가 달라지므로, 저장 데이터에 `mapVersion` 필드를 넣어 생성기 변경 시 올릴 것.

### 5-3. [P1] 스테이지 선택 화면
- 타이틀 → 스테이지 선택 → 플레이 플로우. 50개를 10×5 그리드로, 클리어한 스테이지+1까지 선택 가능.
- 기존 상태 머신(`game.state`: title/play/clear/gameover/win)에 `select` 상태 추가.
- 키보드 화살표 + Enter로 조작 (마우스는 선택 사항).

### 5-4. [P2] 새 테마 (얼음/용암)
- `THEMES`에 항목 추가 + `LevelBuilder` 테마 문자열 + `DECOS`에 장식 세트 추가만으로 확장 가능하게 되어 있다.
- 타일 인덱스는 `assets/tiles.png`(20열)에서 골라 `T`에 추가. `TOP_*` 지형 세트는 [단독, 왼캡, 중앙, 오른캡] 4개 인덱스 규약.
- 얼음 테마는 마찰 감소(`p.vx *= Math.pow(0.0001, dt)` 부분을 테마별 계수로), 용암 테마는 바닥 즉사 타일 등 기믹까지 넣으면 좋음.
- 테마 순환 로직(`(stage-1) % 3`)을 새 테마 포함으로 갱신 — 단, 함정 #2 때문에 **기존 스테이지의 테마가 바뀌는 건 허용**(지형은 안 바뀜), 지형까지 바꾸려면 mapVersion 올릴 것.

### 5-5. [P2] 보스 스테이지 (10/20/30/40/50)
- 제안: 해당 스테이지는 절차 생성 대신 전용 빌더(짧은 아레나 맵) + 보스 엔티티.
- 보스 스펙 초안: 체력 3~5(밟기 3회), 패턴은 돌진↔점프 반복, 피격 시 무적 1초. `ENEMY_BOX`에 `boss` 추가, `enemies` 업데이트 루프에 타입 분기.
- `LEVELS` 배열에서 해당 인덱스만 교체하면 된다: `LEVELS[9] = buildBossLevel1` 식.

### 5-6. [P2] 모바일 터치 입력
- `index.html`에 좌/우/점프 온스크린 버튼(터치 영역), `keys` 객체에 같은 코드(`ArrowLeft` 등)를 흉내 내 주입하는 방식이 최소 침습.
- `touchstart/touchend`로 처리, viewport meta 확인.

### 5-7. [P3] 나머지 (README "다음 개선 아이디어"와 동기화)
- 사운드 음량 조절 UI, 벽 슬라이드/대시, 리플레이, 웹 레벨 에디터.
- 생성 레벨 밸런스 튜닝: 실플레이 피드백 기반으로 `d` 계수들(적 확률 `0.35 + d*0.55`, 톱니 비율 `0.2 + d*0.45`, 구덩이 폭 `2~3+round(d*2)`)만 조정하면 된다. 시드 공식은 건드리지 말 것.
